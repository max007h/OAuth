POC OAuth2 RAR+PAR — Fine-Grained Authorization with PingIdentity

Technical Documentation — Self-hosted Docker Stack | Spring Boot · Angular · PingFederate · PingAuthorize

1. POC Overview

This Proof of Concept demonstrates the implementation of a fine-grained authorization system in a banking context, combining OAuth2 RAR (Rich Authorization Requests — RFC 9396) and PAR (Pushed Authorization Requests — RFC 9126) using the PingIdentity stack deployed self-hosted via Docker.

The goal is to show how business intent (IBAN, amount, transfer type) can be encoded directly into the OAuth2 token at authorization time, and how fine-grained access decisions can be delegated to a centralized Policy Decision Point (PDP).

Tech Stack

Angular (SPA) Spring Boot Spring Security PingFederate (AS) PingAuthorize (PDP) PingDirectory (LDAP) Docker Compose OAuth2 RAR · PAR · PKCE

2. POC Scenarios

# Actor Role Scenario
1 Customer_1 CUSTOMER View own wire transfers (access restricted to their own IBAN)

3 Customer_1 / Customer_2 CUSTOMER Initiate a wire transfer or payment via PAR+RAR (IBAN + amount encoded in the token)
4 Approbateur_1 APPROVER View pending payment requests awaiting validation
5 Approbateur_1 APPROVER Approve or reject a wire transfer exceeding the authorized limit

3. Docker Infrastructure

Docker Compose was chosen as the deployment method for this POC for several reasons. First, the PingIdentity suite requires multiple interdependent services to run simultaneously, and Docker Compose makes it straightforward to define and orchestrate them in a single file. Second, it allows the entire stack to be reproduced consistently across different machines — Windows or macOS — without manual installation of each component. Third, PingIdentity provides official Docker images for all its products, which significantly reduces setup time and configuration effort. Fourth, environment variables and volume mounts make it easy to inject license files and custom configurations without modifying the images. Fifth, the depends_on directive ensures PingFederate and PingAuthorize only start after PingDirectory is ready, which is critical for LDAP connectivity. Finally, this approach closely mirrors a real enterprise deployment, making the POC findings directly transferable to production-grade infrastructure decisions.

The base URL for all API calls is https://localhost:9999/pf-admin-api/v1 with basic auth Administrator / 2FederateM0re. The header X-XSRF-Header: PingFederate is required on all requests.

7. Authorization Detail Processor — Plugin Development & Deployment

⚠️ The Authorization Detail Processor is MANDATORY

PingFederate will not allow saving an Authorization Detail Type without a registered processor. The processor must be compiled as a JAR and deployed into the PingFederate container before it can be selected in the UI.

What is an Authorization Detail Processor?

It is a Java plugin that implements the AuthorizationDetailProcessor interface from the PingFederate SDK. It is called by PingFederate during the OAuth2 PAR+RAR flow to validate, enrich, and describe the authorization_details object submitted by the client.

Step 1 — Extract the PingFederate SDK

The SDK JAR is bundled inside the PingFederate Docker container. Extract it to your local project directory before compiling the plugin:

# List available SDK JARs inside the container
docker exec ping-pingfederate-1 ls /opt/out/instance/server/default/lib/

# Copy the SDK JAR to your local project (Windows)
docker cp ping-pingfederate-1:/opt/out/instance/server/default/lib/pingfederate-sdk.jar ^
  "D:\Docker Imgs\Ping\Plugin NoProcessor\pingfederate-sdk.jar"

Step 2 — Build the Plugin

The plugin project is located at D:\Docker Imgs\Ping\Plugin NoProcessor. Build it using Maven with the extracted SDK as a local dependency:

cd "D:\Docker Imgs\Ping\Plugin NoProcessor"
mvn clean package

# Verify the compiled .class files are present in the JAR
jar tf target\banking-transfer-processor-1.0.jar | findstr ".class"

Step 3 — Deploy the JAR into PingFederate

# Windows
docker cp target\banking-transfer-processor-1.0.jar ^
  ping-pingfederate-1:/opt/out/instance/server/default/deploy/

# Mac
docker cp target/banking-transfer-processor-1.0.jar \
  env-pingfederate-1:/opt/out/instance/server/default/deploy/

# Restart PingFederate to load the new plugin
docker restart ping-pingfederate-1
# or on Mac:
docker restart env-pingfederate-1

Step 4 — Register in Admin UI

Once deployed and PingFederate restarted, go to System → OAuth Settings → Authorization Detail Types → Add and select Banking Transfer Processor from the processor dropdown.

Plugin Methods — BankingTransferProcessor.java

MethodWhen CalledRole
configure() Plugin startup Loads configuration from Admin UI. Empty in this POC — no config params needed.
getPluginDescriptor() Plugin registration Returns plugin metadata to PingFederate: name "Banking Transfer Processor", version "1.0", GUI fields.
validate() Authorization time (PAR request) Validates the RAR payload sent by Angular. Returns valid in this POC. Can be extended to check amount > 0, valid IBAN format, currency whitelist.
enrich() After validation Enriches the authorization_details object before it is stored in the JWT. Returns detail as-is in this POC.
getUserConsentDescription() Consent screen display Returns human-readable string shown to the user: "Transfer of 150.00 EUR to IBAN FR76... | Reference: rent". Reads: amount, currency, iban, label from detail.getDetail().
isEqualOrSubset() Token reuse / incremental auth Checks if a new authorization request is covered by an existing grant. Returns true in this POC.

Advanced Controls — External API Calls from Plugin Methods

For more complex business rules involving entity relationships — such as verifying that a creditor IBAN belongs to a known counterparty, checking account balance limits, or calling a fraud detection system — the processor methods can make HTTP calls to external APIs directly from Java. This avoids duplicating validation logic in Spring Boot.

💡 Example: In validate(), call GET /api/accounts/{iban}/status to verify the creditor IBAN is not blocked before issuing the token. In enrich(), call GET /api/accounts/{iban}/details to resolve the IBAN to a beneficiary name and inject it directly into the JWT — so Spring Boot receives it without a second lookup.

Use CaseMethodAPI Call Example
Validate creditor IBAN exists validate() GET /api/accounts/{iban}/status
Check transfer amount vs. daily limit validate() GET /api/customers/{uid}/limits
Resolve IBAN to beneficiary name enrich() GET /api/accounts/{iban}/details
Add fraud risk score to token enrich() POST /api/fraud/score
Check if grant covers new request isEqualOrSubset() Local comparison — no API needed

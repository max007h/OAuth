POC OAuth2 RAR+PAR — Fine-Grained Authorization with PingIdentity

Technical Documentation — Self-hosted Docker Stack | Spring Boot · Angular · PingFederate · PingAuthorize

1. POC Overview

This Proof of Concept demonstrates the implementation of a fine-grained authorization system in a banking context, combining OAuth2 RAR (Rich Authorization Requests — RFC 9396) and PAR (Pushed Authorization Requests — RFC 9126) using the PingIdentity stack deployed self-hosted via Docker.

The goal is to show how business intent (IBAN, amount, transfer type) can be encoded directly into the OAuth2 token at authorization time, and how fine-grained access decisions can be delegated to a centralized Policy Decision Point (PDP).

Tech Stack

Angular (SPA) Spring Boot Spring Security PingFederate (AS) PingAuthorize (PDP) PingDirectory (LDAP) Docker Compose OAuth2 RAR · PAR · PKCE

2. POC Scenarios

# Actor Role Scenario
1 Customer_1 CUSTOMER View own wire transfers (access restricted to their own IBAN)
2 Customer_2 CUSTOMER View own wire transfers (access restricted to their own IBAN)
3 Customer_1 / Customer_2 CUSTOMER Initiate a wire transfer or payment via PAR+RAR (IBAN + amount encoded in the token)
4 Approbateur_1 APPROVER View pending payment requests awaiting validation
5 Approbateur_1 APPROVER Approve or reject a wire transfer exceeding the authorized limit

3. Docker Infrastructure

Docker Compose was chosen as the deployment method for this POC for several reasons. First, the PingIdentity suite requires multiple interdependent services to run simultaneously, and Docker Compose makes it straightforward to define and orchestrate them in a single file. Second, it allows the entire stack to be reproduced consistently across different machines — Windows or macOS — without manual installation of each component. Third, PingIdentity provides official Docker images for all its products, which significantly reduces setup time and configuration effort. Fourth, environment variables and volume mounts make it easy to inject license files and custom configurations without modifying the images. Fifth, the depends_on directive ensures PingFederate and PingAuthorize only start after PingDirectory is ready, which is critical for LDAP connectivity. Finally, this approach closely mirrors a real enterprise deployment, making the POC findings directly transferable to production-grade infrastructure decisions.

Manual Configuration Checklist

Once logged in, configure the following components in order:

#ComponentLocation in UIDetails
1 Data Store System → Data Stores LDAP connection to PingDirectory on localhost:1636 — bind DN cn=administrator
2 Password Credential Validator System → Password Credential Validators LDAP Username Password Validator using the Data Store above
3 OAuth Client OAuth Server → Clients Client ID: banking-spa — public client — PAR required — redirect http://localhost:4200
4 Scopes OAuth Server → Scope Management openid, payment:read, banking_transfer
5 Attribute Mappings OAuth Server → Access Token Manager Map ibanAccount, groups (isMemberOf), authorization_details into JWT
6 PAR Policy OAuth Server → Authorization Server Settings Enable Pushed Authorization Requests — set as required for banking-spa
7 RAR Type OAuth Server → Authorization Detail Types Type: banking_transfer
8 PDP Connection System → External Services Connect PingFederate to PingAuthorize on https://localhost:7443

6. PingFederate Configuration via REST API

All PingFederate configuration can be done programmatically via the Admin REST API instead of the UI. This approach is reproducible, scriptable, and can be version-controlled. The base URL for all API calls is https://localhost:9999/pf-admin-api/v1 with basic auth Administrator / 2FederateM0re. The header X-XSRF-Header: PingFederate is required on all requests.

Step 1 — Data Store (LDAP connection to PingDirectory)

curl -k -u Administrator:2FederateM0re \
  -X POST \
  -H "Content-Type: application/json" \
  -H "X-XSRF-Header: PingFederate" \
  https://localhost:9999/pf-admin-api/v1/dataStores \
  -d '{
    "type": "LDAP",
    "name": "PingDirectory-LDAP",
    "ldapType": "PING_DIRECTORY",
    "useSsl": true,
    "hostnames": ["pingdirectory"],
    "port": 1636,
    "baseDn": "dc=example,dc=com",
    "bindAnonymously": false,
    "userDN": "cn=administrator",
    "password": "2FederateM0re"
  }'

Step 2 — Password Credential Validator

curl -k -u Administrator:2FederateM0re \
  -X POST \
  -H "Content-Type: application/json" \
  -H "X-XSRF-Header: PingFederate" \
  https://localhost:9999/pf-admin-api/v1/passwordCredentialValidators \
  -d '{
    "id": "ldapPCV",
    "name": "LDAP PCV",
    "pluginDescriptorRef": {
      "id": "org.sourceid.saml20.domain.LDAPUsernamePasswordCredentialValidator"
    },
    "configuration": {
      "tables": [{
        "name": "Authentication Error Overrides"
      }],
      "fields": [
        { "name": "LDAP Datastore", "value": "PingDirectory-LDAP" },
        { "name": "Search Base", "value": "ou=People,dc=example,dc=com" },
        { "name": "Search Filter", "value": "uid=${username}" }
      ]
    }
  }'

Step 3 — OAuth Client (banking-spa)

curl -k -u Administrator:2FederateM0re \
  -X POST \
  -H "Content-Type: application/json" \
  -H "X-XSRF-Header: PingFederate" \
  https://localhost:9999/pf-admin-api/v1/oauth/clients \
  -d '{
    "clientId": "banking-spa",
    "name": "Banking SPA",
    "grantTypes": ["AUTHORIZATION_CODE"],
    "redirectUris": ["http://localhost:4200"],
    "requirePushedAuthorizationRequests": true,
    "pkceEnforcement": "REQUIRED",
    "allowAuthenticationApiInit": false,
    "scopes": ["openid","payment:read","banking_transfer"],
    "clientAuthType": "NONE"
  }'

Step 4 — RAR Authorization Detail Type

curl -k -u Administrator:2FederateM0re \
  -X POST \
  -H "Content-Type: application/json" \
  -H "X-XSRF-Header: PingFederate" \
  https://localhost:9999/pf-admin-api/v1/oauth/authorizationDetailTypes \
  -d '{
    "type": "banking_transfer",
    "description": "Banking wire transfer authorization details"
  }'

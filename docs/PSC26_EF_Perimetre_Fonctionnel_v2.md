# PSC26 — Tableau du Périmètre Fonctionnel Couvert

> **Document** : Dossier d'Expression de Besoins — Benchmark DAST/IAST  
> **Classification** : C2 Restreint  
> **Projet** : PSC26 — Contrôles Sécurité APIs — gLBP

---

| ID | Intitulé | Description | Priorité | Réf. DORA/PTS |
|---|---|---|---|---|
| EF-01 | Découverte & couverture de la surface d'attaque | La solution doit être capable de découvrir et de cartographier la surface d'attaque de toute application ou service à scanner, selon deux modes complémentaires : **Mode application web** : crawling automatique de l'application (navigation dans les pages HTML, détection des formulaires, des liens, des redirections, des mécanismes d'authentification basés sur session ou cookie). Le crawler doit être capable de suivre les flux de navigation multi-étapes (ex : tunnel de paiement, workflow de validation) et de rejouer des séquences enregistrées via HAR. **Mode API** : découverte des endpoints à partir d'une spécification OpenAPI/Swagger (2.x et 3.x), WSDL (SOAP), GraphQL SDL ou AsyncAPI. En l'absence de spécification, la solution doit effectuer une découverte active des endpoints exposés. Dans les deux cas, la solution doit identifier les ressources non documentées ou non référencées (endpoints shadow, méthodes HTTP non déclarées, paramètres cachés). | Haute | RTS 11.2.B |
| EF-02 | Scan DAST automatisé CI/CD | La solution doit exécuter des analyses de sécurité dynamiques (DAST) de façon automatique à chaque déclenchement du pipeline GitLab CI/CD, sur l'environnement de staging ou d'intégration. Le scan doit couvrir aussi bien les **applications web** (rendu HTML, formulaires, sessions) que les **APIs** (REST, SOAP, GraphQL) exposées par l'application cible. Il doit : couvrir l'ensemble des catégories de l'OWASP Top 10 Web (2021) et de l'OWASP API Security Top 10 (2023) selon le type de cible ; s'appuyer sur la spécification OpenAPI lorsqu'elle est disponible pour les composants API, et sur le crawling pour les composants web ; bloquer automatiquement le pipeline si une vulnérabilité de niveau CRITICAL est détectée ; publier les résultats au format SARIF dans le GitLab Security Dashboard. | Haute | RTS 10.2.A / RTS 10.2.B |
| EF-03 | Scan DAST On-Demand | La solution doit permettre le déclenchement de scans de sécurité dynamiques de façon planifiée ou manuelle sur les environnements de preprod et de production. Elle doit couvrir les applications web et les APIs exposées sur le périmètre cible. Elle doit proposer : un **mode passif** (observation du trafic réel sans envoi de payload actif, zéro risque de perturbation) ; un **mode actif limité** (avec rate limiting strict, circuit breaker configurable, et plage horaire de scan définie hors heures de pic métier). Les applications et APIs classifiées FCI doivent faire l'objet d'un scan hebdomadaire automatique conformément à DORA RTS 10.2.B. Le rapport de scan doit être produit de façon asynchrone et transmis au CSIRT et au RSSI au format PDF/HTML classifié C2 Restreint. | Haute | RTS 10.2.B |
| EF-04 | IAST multi-langages | La solution doit instrumenter les applications de l'intérieur pendant la phase de tests du pipeline CI/CD (BUILD) afin de détecter les vulnérabilités par analyse du flux d'exécution réel. L'instrumentation doit s'effectuer **sans aucune modification du code source** applicatif. Elle doit couvrir l'ensemble des langages du parc applicatif : Java (JVM 8+), Python (3.8+), Node.js (16+), .NET Core (6+), Go et PHP. La détection repose sur le **taint tracking** : suivi du flux de données depuis l'entrée (paramètre HTTP, body JSON/XML, header, cookie, champ de formulaire) jusqu'au sink dangereux (requête SQL, exécution système, écriture fichier, réponse HTTP). L'IAST s'applique indifféremment aux composants web et aux composants API de l'application instrumentée. Chaque finding doit correspondre à un chemin d'exécution réellement parcouru pendant les tests, éliminant structurellement les faux positifs liés à du code non exécuté. | Haute | RTS 10.2.A |
| EF-05 | Contrôles de durcissement | La solution doit vérifier la conformité des configurations de sécurité des applications et des APIs scannées par rapport aux standards de durcissement définis dans **PTS-MIA-PR-HARDEN**. Les points de contrôle obligatoires incluent : présence et valeur correcte des headers de sécurité HTTP (HSTS, CSP, X-Content-Type-Options, X-Frame-Options, Referrer-Policy) ; version et configuration TLS (1.2 minimum, 1.3 recommandé) ; mécanismes d'authentification en place et correctement configurés (absence de credentials par défaut, politique de mots de passe, gestion des sessions) ; absence d'exposition de données sensibles dans les réponses (hash de mot de passe, token, données personnelles, informations techniques internes). Les écarts détectés doivent être remontés comme findings avec leur niveau de sévérité CVSS et leur référence CWE. | Haute | PTS-MIA-PR-HARDEN / RTS 10.4.D |
| EF-06 | Gestion des faux positifs persistante | La solution doit proposer un workflow structuré de requalification des vulnérabilités détectées avec les statuts suivants : **Open**, **Confirmed**, **False Positive**, **Accepted Risk**, **Fixed**, **In Progress**. Les règles suivantes s'appliquent : la requalification en **False Positive** est réservée exclusivement à l'équipe sécurité, avec justification textuelle obligatoire et horodatage ; un finding requalifié en False Positive ne remonte plus dans les scans suivants ; chaque entrée de la baseline de FP doit avoir une durée de vie maximale configurable (90 jours recommandés), passé laquelle le finding redevient actif pour re-validation ; tout abus de requalification (volume anormal sur une courte période) doit déclencher une alerte vers le RSSI. L'ensemble des actions de requalification doit être tracé, horodaté et non falsifiable. | Haute | RTS 10.2.B |
| EF-07 | Rapports personnalisables | La solution doit générer des rapports de scan dans les formats suivants : **SARIF** (intégration native GitLab Security Dashboard) ; **JSON brut** (alimentation VOC et SIEM) ; **HTML/PDF** personnalisé (destiné au CSIRT, RSSI et comités d'audit), incluant le logo de l'établissement, la classification C2 Restreint et le contexte projet ; **CSV** (suivi qualité et consolidation). Un **rapport différentiel** doit être disponible nativement, ne listant que les nouveaux findings apparus depuis le scan précédent. Les rapports doivent pouvoir être filtrés par niveau de sévérité, par projet, par type de cible (application web ou API) ou par environnement. | Haute | DICT C:3 |
| EF-08 | Alimentation VOC & indicateurs MCS | La solution doit exporter les résultats de scan vers le **VOC (Vulnerability Operations Center)** pour agrégation et priorisation. Chaque finding exporté doit inclure : le score CVSS v3.1, le score EPSS (probabilité d'exploitation à 30 jours), le flag KEV (Known Exploited Vulnerabilities — CISA) si applicable, et le TRS. La solution doit produire les indicateurs opérationnels suivants pour le **Maintien en Condition de Sécurité (MCS)** : nombre de vulnérabilités actives par criticité, taux de faux positifs, taux de couverture des scans, nombre de points de contrôle déployés et conformes, évolution temporelle du score de sécurité par application. | Haute | RTS 2.2.C |
| EF-09 | Fiabilisation de l'inventaire applicatif | La solution doit contribuer à la fiabilisation de l'inventaire des applications et services exposés en alimentant les données de cartographie issues des scans. Elle doit couvrir : le **flux** (nouveaux projets en cours de développement, intégration dans le cycle DevSecOps dès la phase de build) et le **stock** (applications et APIs déjà déployées en production, découverte de l'existant non référencé). La solution doit être compatible avec les applications déjà en place sans nécessiter de refonte de l'existant. Les données produites doivent permettre de maintenir un inventaire à jour, fiable et exploitable par les équipes sécurité et architecture. | Haute | RTS 8.6 / RTS 11.2.B |
| EF-10 | Traçabilité & auditabilité | La solution doit journaliser l'ensemble des actions de scan, de requalification et d'administration avec les informations suivantes : horodatage, identité de l'utilisateur, action réalisée, résultat. Les logs doivent être conservés pendant une durée minimale de **1 an** conformément aux exigences DICT (T:3). Chaque rapport de scan doit inclure le lien vers le pipeline GitLab ayant déclenché le scan (branche, commit SHA, environnement). Les données d'audit doivent être exportables à la demande du CSIRT pour inspection ou audit de conformité DORA. | Haute | DICT T:3 / DORA |
